- [ ] cloud substructure shifts visibly as the camera translates, in ways that read as "structural transitions" rather than smooth motion. Investigated extensively with the cloud pixel inspector (`bevy_pbr_clouds_planet::inspect` + Atmosphere → Inspector sub-tab); current best understanding is that it's **expected behaviour of perspective raymarched volumetric noise** rather than a localised bug. Mechanism: each pixel's ray has a unique direction, so under lateral camera motion the per-pixel sample-position shift is non-uniform (`Δsample = Δcam − (Δcam · ray) · ray`); combined with high-frequency noise (~60 m mip-2 texels), neighbouring pixels cross feature boundaries at different camera positions but with spatial coherence, so the field looks like it's restructuring rather than translating. Confirmed via the inspector that for fixed camera the values are bit-exactly deterministic, and the low-frequency components (`climate_base`, `regional_coverage`, `v_profile`) are stable while only the noise-driven `shape` term shifts; ruled out as the cause: temporal pass / history, per-pixel hash, per-world-cell hash, sub-grid jitter (still present at magnitude 0), `noise_lod_bias`, `primary_step_world_m`, the world-snap chord boundary (boundary fade made this smooth), CPU/shader floating-origin precision mismatch, climate sim, the `k_first` integer, the density-smoothstep band width (slider exposed, no effect on transitions), and shape saturation (`saturate` is mathematically a no-op since the remap output is already in [0, 1]). In normal use (temporal pass + denoise on at defaults), the temporal accumulation hides this; the artefact only surfaces when all noise-amelioration is disabled for diagnostic purposes. Possible fixes if it ever needs proper attention: multi-tap sub-position averaging of the noise per primary-march cell (4–8× sampling cost), or motion-vector-based temporal reprojection that uses the actual per-sample integrated depth instead of the shell-midpoint heuristic.
